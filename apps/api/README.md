# API App

Backend service boundary for authoritative game state, domain orchestration, realtime fan-out, and worker entry points.

The runtime stack target is NestJS + Prisma + PostgreSQL. Phase 2 now includes a real NestJS bootstrap, platform runtime modules under `src/platform`, Prisma datasource setup, and a non-gameplay `/api/health` endpoint.

Live gameplay runtime behavior currently exists for `player`, `crime`, `jail`, `hospital`, `inventory` including basic equipment state, `market` including fixed-price player trading, `combat`, `gangs` including basic invites and leader-only permissions, `territory` including leader-authorized district claims, cooldown-based district payouts, and minimal district wars, `missions` with starter accept-progress-complete flows, `achievements` with static catalog entries and event-driven unlock tracking, `notifications` with a persistent player activity feed backed by explicit domain events, `leaderboard` with static public player rankings backed by persisted state, `seasons` with persistent season records and single-active-season lifecycle control, and a narrow `admin-tools` balance surface for inspecting and updating crime, district, and starter shop values through explicit endpoints. Crime balance values and starter shop prices now persist as explicit balance records and successful admin balance updates now write minimal audit log entries. The admin surface is now role-protected through persisted account admin flags instead of shared-secret-only access. The API now also includes a small synchronous in-process domain event layer for decoupling module progress updates from direct gameplay service calls. Other gameplay modules under `src/modules` remain scaffolds until their owning features are implemented.

The API now also includes a minimal `auth` runtime module for email/password account creation, bearer-session login, `GET /api/auth/me`, one-account-to-one-player ownership, persisted `isAdmin` account roles, and a narrow first wave of authenticated `/api/me/*` gameplay endpoints. Most legacy `players/:playerId/...` routes remain transitional and are not yet fully ownership-hardened.
